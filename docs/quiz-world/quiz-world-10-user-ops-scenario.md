# 通知型早押しクイズワールド 初期10人運用シナリオ

## 目的

Season 0の10人テストで、通知型早押しクイズの面白さ、安全性、ランク理解、運用負荷を検証する。

## 参加者募集

### 条件

- 18歳以上。
- 7日間のテストに参加できる。
- 画面内通知を確認できる。
- クイズを出す側と答える側の両方を試せる。
- 不具合や感想を共有できる。

### 募集時に伝えること

- MVPは招待制。
- Phase 1は画面内通知で、Web Pushはまだ使わない。
- 1日数回クイズが届く可能性がある。
- quiet hoursと通知上限がある。
- 不適切な問題は通報できる。
- テスト終了後にアンケートを行う。

## 招待コード配布

1. adminが招待コードを発行する。
2. 参加予定者に個別に配布する。
3. 有効期限を設定する。
4. 使用済みコードを確認する。
5. 10人到達後は新規参加を止める。

一般ユーザーによる招待はSeason 0では行わない。

## 10人参加確認

adminは以下を確認する。

- world.member_limit = 10。
- active world_members = 10。
- 18歳以上確認が全員完了。
- 規約同意が全員完了。
- 通知設定の初期値が入っている。
- suspended userがいない。

## waitlist確認

10人到達後に来た希望者はwaitlistへ入れる。

確認する項目。

- email
- display_name
- created_at
- status
- 招待済みかどうか

Season 0中は、原則としてwaitlistから自動参加させない。

## 1日目のテスト

### 目的

登録、出題、画面内通知、回答、結果表示の基本体験を確認する。

### 目標

- 全員がログインできる。
- 少なくとも5問出題される。
- 各ユーザーが1回以上回答する。
- start_at前に問題が見えないことを確認する。
- 結果画面で順位が理解される。

### admin確認

- 招待コード使用状況。
- quiz_launches作成状況。
- quiz_recipients作成状況。
- answers作成状況。
- エラー報告。

## 3日目のテスト

### 目的

通知頻度、ランク、評価導線、安全導線を確認する。

### 目標

- 1日平均5問以上出題されている。
- 評価が複数件入っている。
- 出題者スコアと回答者スコアが変動している。
- quiet hoursや通知上限が理解されている。

### 低評価/通報テスト

安全な範囲で、以下のテスト用問題を用意する。

- 答えが曖昧な問題。
- 難しすぎる問題。
- 簡単すぎる問題。

「不適切」タグや通報導線が見つけやすいか確認する。
実際に不快な内容は使わない。

## 7日目のテスト

### 目的

継続性、ランク理解、参加枠拡大判断の材料を集める。

### 目標

- 少なくとも7日続く。
- 参加者の半数以上が3日以上参加。
- 1日平均5問以上出題。
- 回答順位の納得感を確認。
- 評価/通報導線が実際に使われる。
- ランク上昇が理解される。
- 「もう少し続けたい」という反応がある。

## 問題数の目標

| 期間 | 目標 |
| --- | --- |
| 1日目 | 5問以上 |
| 3日目まで | 累計15問以上 |
| 7日目まで | 累計35問以上 |

出題数が少ない場合は、出題者ランクや出題導線のわかりにくさを確認する。

## 通知頻度

Phase 1は画面内通知で検証する。

初期目安。

- 通常モード: 1日最大5問。
- 集中モード: 1日最大20問。
- 休憩モード: 通知なし。
- quiet hours: 22:00〜8:00。

確認すること。

- 多すぎると感じるか。
- 少なすぎると感じるか。
- `/home` を開く動機になるか。
- Phase 2でWeb Pushが必要か。

## 出題者ランク検証

確認すること。

- 良問評価でスコアが上がることが理解されるか。
- 微妙評価や曖昧タグで下がることが納得されるか。
- 配信人数が増える仕組みが期待になるか。
- 課金ではなく評価で伸びる方針が受け入れられるか。

## 回答者ランク検証

確認すること。

- 正解者順位がランクに効くことが理解されるか。
- 早い不正解が大きく加点されないことが納得されるか。
- 未回答ペナルティなしが気軽さにつながるか。
- 難問正解ボーナスが楽しいか。

## admin対応のテスト

adminが実際に確認する操作。

- 通報一覧を見る。
- 通報詳細を見る。
- question.statusをsuspendedにする。
- user/profile.statusをsuspendedにするテストを行う。
- waitlistを見る。
- 招待コードを追加発行する。
- member_limitを変更しない状態で確認する。
- admin_audit_logsが残ることを確認する。

本物の不適切投稿を待たず、テスト用データで安全に確認する。

## アンケート項目

### 基本

- 7日間で何日参加したか。
- クイズを出したか。
- クイズに回答したか。
- 画面内通知に気づけたか。

### 体験

- いつ届くかわからない感覚は面白かったか。
- 通知頻度は多すぎたか、少なすぎたか。
- start_atの15秒待ちは適切だったか。
- 回答時間60秒は適切だったか。
- 順位は納得できたか。
- 結果画面はわかりやすかったか。

### ランク

- 出題ランクの意味はわかったか。
- 回答ランクの意味はわかったか。
- ランクを上げたいと思ったか。

### 安全

- 評価導線は使いやすかったか。
- 通報導線は見つけやすかったか。
- 不快な通知や問題があったか。
- ブロック機能は必要そうか。

### 継続

- もう少し続けたいか。
- 15人に増えたら参加したいか。
- Web Pushが欲しいか。
- ギルドが欲しいか。

## 継続判断基準

10人テストの成功条件。

- 少なくとも7日続く。
- 参加者の半数以上が3日以上参加。
- 1日平均5問以上出題。
- 通知が不快すぎない。
- 回答順位が納得できる。
- 評価/通報導線が使われる。
- ランク上昇が理解される。
- もう少し続けたいという反応がある。

## Season 1へ進む判断

Season 1で15人へ増やす前に確認する。

- 通報率が高すぎない。
- admin対応が回る。
- 出題数が維持されている。
- 回答参加率が落ちすぎていない。
- Web Pushなしでも最低限遊べる。
- Web Pushが必要ならPhase 2計画を先に立てる。
