# 通知型早押しクイズワールド コンセプト

## 位置づけ

「通知型早押しクイズワールド」は、既存の Smart Buzzer とは別企画として扱う。
本ドキュメントは企画・設計フェーズの整理であり、既存アプリコード、production 環境、Stripe、Vercel、Supabase、env、legal page、cleanup には触れない。

## サービス概要

通知型早押しクイズワールドは、少人数招待制で始める、非同期型の早押しクイズゲームである。

- 24時間365日、いつクイズが届くかわからない。
- 誰かがクイズを出すと、選ばれたプレイヤーに通知が届く。
- 通知を受け取ったプレイヤーは、指定された開始時刻から問題を開き、他のプレイヤーより早く回答する。
- 回答後は、全回答者の正誤、回答順位、正解者順位を確認できる。
- 回答者はクイズの内容を評価できる。
- 良問評価が多い出題者は出題ランクが上がる。
- 正解率、速さ、難問正解などで回答者は回答ランクが上がる。

MVPでは「SNS」ではなく、「少人数招待制の通知型早押しクイズゲーム」として成立させることを優先する。

## なぜ普通のSNSではないのか

一般的なSNSは、タイムライン、コメント、DM、フォローによって継続接触を作る。一方、この企画の中心体験は「通知が来た瞬間にゲームが始まるかもしれない」という緊張感にある。

普通のSNSに寄せすぎると、次の問題が起きやすい。

- クイズより投稿や会話が主役になる。
- 通知が単なる更新通知になり、ゲーム体験として弱くなる。
- コメントやDMのモデレーション負荷が増える。
- 初期10人の小さな検証で、見るべき指標が分散する。

そのためMVPでは、タイムライン、コメント、DM、フォローを入れず、通知、早押し、結果、評価、ランクに体験を集中させる。

## 1ワールド制にする理由

MVPでは全員が同じ1つのワールドにいる。

理由は次の通り。

| 理由 | 説明 |
| --- | --- |
| 密度を保つ | 初期人数が少ないため、分散させるとクイズが届く頻度が下がる。 |
| 検証しやすい | 出題数、回答率、評価、通知疲れを1つの母集団で見られる。 |
| 世界観を作りやすい | 「小さなクイズ村」が成長していく感覚を出せる。 |
| 運用しやすい | 通報、招待、参加枠管理を最小構成で始められる。 |
| 将来拡張しやすい | 内部的に worlds / world_members を使えば、後から複数ワールドやギルドへ拡張できる。 |

## 初期10人限定にする理由

最初はSNS全体で10人しか参加できない設計にする。

これは制限ではなく、ゲームの世界観と検証品質を高めるための設計である。

- 誰が出したクイズか、誰がよく答えるかを覚えやすい。
- 通知頻度を管理しやすい。
- 通報や不適切クイズの運用を小さく始められる。
- ランクや参加枠の成長が目に見える。
- 初期ユーザーに「自分たちがワールドを育てている」感覚が生まれる。

## 参加枠が増える世界観

ワールドはシーズンごとに成長し、参加枠が増える。

| シーズン | 参加枠 | 世界観 |
| --- | ---: | --- |
| Season 0 | 10人 | 小さなクイズ村。顔ぶれを覚えられる段階。 |
| Season 1 | 15人 | 新しい住人が加わり、出題傾向が広がる段階。 |
| Season 2 | 20人 | ランク上位者が現れ、ギルド構想の準備が始まる段階。 |
| Season 3 | 30人 | 通知設計とランキングが本格的に機能する段階。 |
| Season 4 | 50人 | ギルドやイベントを検討できる規模。 |

参加枠は課金で買うものではない。ワールド全体の活動、クイズ品質、安全性、継続性によって解放される。

## 小さなクイズ村が育つ感覚

初期体験では、巨大なSNSではなく「小さなクイズ村」を目指す。

- 今日は誰が出題したかがわかる。
- 誰が早く正解したかが話題になる。
- 良問を作る人が尊敬される。
- 難問に強い人が目立つ。
- 通知が来るたびに、村の中で小さなイベントが起きる。

この感覚を作るため、MVPでは大規模な公開投稿や自由な会話機能を入れない。

## 出題者ランクと回答者ランクを分ける理由

このゲームには、良いクイズを作る楽しさと、速く正確に答える楽しさの2つがある。

| ランク | 役割 | 成長による効果 |
| --- | --- | --- |
| 出題ランク | 良問を作り、他のプレイヤーに良い体験を届ける | 1日の出題回数、1回の配信人数が増える |
| 回答ランク | 速く、正確に、難問に答える | 称号、ランキング、将来のギルド作成条件などに反映 |

出題者ランクと回答者ランクを分けることで、クイズ作成が得意な人と回答が得意な人の両方が活躍できる。

## 通知がゲーム体験そのものである

このサービスでは、通知は補助機能ではない。通知そのものがゲーム開始の合図であり、体験の中心である。

- いつ届くかわからない緊張感がある。
- 通知を開くとカウントダウンが始まる。
- start_at まで待つことで、端末差や通知到着差をある程度吸収する。
- 回答はサーバー受信順で順位が決まる。
- 結果画面で、誰が早く正解したかが見える。

## 通知疲れを防ぐ必要性

通知が体験の中心であるほど、通知疲れを防ぐ設計が重要になる。

MVPでは次を必須にする。

- quiet hours を設定できる。
- 深夜通知はデフォルトOFFにする。
- 1日の通知上限を設定する。
- 通知モードを選べる。
- 休憩モードを用意する。
- 出題者ランクによって配信人数を制限する。
- 通知対象抽選では、同じ人に偏りすぎないようにする。

通知を増やすことより、通知が届いたときに「参加したい」と思える品質を維持する。

## 将来的なギルド構想

MVPではギルドを入れない。ただし、将来的に小集団、チーム、対抗戦へ拡張できるように設計する。

ギルドは次のような機能として検討する。

- ギルド内ランキング
- ギルド専用クイズ
- ギルド招待
- ギルド対抗戦
- ギルド称号
- ギルド貢献度
- ギルド作成権

内部データモデルでは将来 guilds / guild_members / guild_invites を追加できる余地を残す。

## MVPでSNS機能を最小限にする理由

MVPで入れない機能は次の通り。

- タイムライン
- コメント
- DM
- フォロー
- 自由チャット
- 公開掲示板

理由は明確である。

- 通知型早押しクイズの面白さを先に検証するため。
- 初期10人ではSNS機能よりゲーム密度が重要なため。
- UGCと会話機能を同時に持つと安全運用が重くなるため。
- 通報、ブロック、通知設定、ランキングの基本を先に固めるため。
