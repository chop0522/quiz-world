# 通知型早押しクイズワールド インフラ計画

## 前提

本ドキュメントは新規企画「通知型早押しクイズワールド」のインフラ設計案である。
既存の Smart Buzzer とは別企画として扱い、既存の production、Stripe、Vercel、Supabase、env、legal page、cleanup には触れない。

この段階では、cloud接続、課金設定、DB migration、Vercel env変更、Supabase設定変更、live key切り替えは行わない。

## MVPインフラ案

| 項目 | 候補 |
| --- | --- |
| フロントエンド | Next.js PWA |
| 認証 | Supabase Auth |
| DB | Supabase Postgres |
| アクセス制御 | Supabase RLS |
| リアルタイム | Supabase Realtime |
| ホスティング | Vercel |
| 通知 | Web Push |

## この構成を選ぶ理由

- 10人テストならPWAで十分に検証できる。
- 少人数招待制と相性がよい。
- Next.jsで画面とAPIを素早く作れる。
- Supabase Auth / Postgres / RLS により、認証とデータ制御をまとめやすい。
- Supabase Realtime で結果更新やステータス更新を扱いやすい。
- Vercel でMVPを小さく公開しやすい。
- 既存の Smart Buzzer で得た技術知見を活用できる。

ただし、既存 Smart Buzzer の環境や設定を流用・変更することはしない。必要な場合は別プロジェクト、別環境として明確に分ける。

## PWA通知の注意点

通知がサービスの中心であるため、PWA通知の制約はMVP前に把握しておく。

- ユーザーが通知許可を出す必要がある。
- ブラウザやOSによって挙動が異なる。
- iOSではホーム画面追加などの条件がある。
- 通知到着の正確性はネイティブアプリより弱い可能性がある。
- 端末や省電力設定の影響を受ける。

MVPでは「通知型早押しが面白いか」を10人で検証することを優先し、本格化したら Expo / React Native を検討する。

## 実装上の重要原則

- 順位判定はサーバー主導にする。
- クライアント時刻は順位判定に使わない。
- 出題対象抽選はサーバー側で行う。
- 通知対象はクライアントから改ざんできないようにする。
- quiz_recipients に存在するユーザーだけが回答できる。
- RLSでユーザーごとのアクセスを制御する。
- 管理者操作は通常ユーザー操作と明確に分ける。
- 管理者操作には監査ログを検討する。
- cloud操作や課金はMVP初期には入れない。
- Stripeによる配信人数増加は設計しない。

## RLS前提のアクセス制御

| データ | 基本方針 |
| --- | --- |
| プロフィール | 本人は自分の設定を更新可能。必要な公開情報のみ他者に公開。 |
| ワールド所属 | 本人は自分の所属状態を参照可能。管理者は全体管理可能。 |
| クイズ | 作成者は自分のクイズを管理可能。配信済み問題の編集は制限。 |
| 配信 | 出題者、対象者、管理者に必要範囲を公開。 |
| 回答 | 本人が回答作成。結果公開後は対象者に順位情報を公開。 |
| 通報 | 通報者が作成。管理者が対応。 |
| ブロック | 本人のブロック関係のみ管理可能。 |

## 通知処理の段階設計

| 段階 | 実装方針 |
| --- | --- |
| MVP初期 | サーバー処理で対象抽選し、Web Push送信。 |
| MVP後半 | notification_logs を使って送信品質を確認。 |
| 30人以上 | キューを追加し、送信と回答受付を分離。 |
| 本格版 | Pushプロバイダ、再送、失敗分類、レート制御を整備。 |
