# 通知型早押しクイズワールド MVP仕様

## 前提

本仕様は、既存の Smart Buzzer とは別企画として作成する。
MVPでは実装、DB migration、cloud data 変更、Stripe 操作、Vercel env 変更、Supabase 既存設定変更、cleanup、live key 切り替えは行わない。

MVPの目的は、「少人数招待制の通知型早押しクイズゲーム」として面白さと安全性を検証することである。

## MVPの基本方針

- 最初は1つのワールド制にする。
- 初期参加上限は10人にする。
- 満員時はウェイトリストへ誘導する。
- 18歳以上限定にする。
- タイムライン、コメント、DM、フォローは入れない。
- 課金は入れない。
- AI自動判定に頼らない。
- クイズ形式は四択を優先する。
- 通知、カウントダウン、回答、結果、評価、ランクを中核体験にする。

## MVPに含める機能

### アカウント・参加

- ユーザー登録
- ログイン
- 18歳以上確認
- 利用規約同意
- プライバシーポリシー同意
- 1ワールド制
- 初期参加上限10人
- 満員時のウェイトリスト
- 招待制または管理者承認制
- プロフィール

### クイズ体験

- クイズ作成
- クイズ出題
- 通知対象の抽選
- 通知
- 開始カウントダウン
- 回答
- 回答順位表示
- 正解者順位表示
- 全体結果表示
- クイズ評価

### 成長・制限

- 出題者ランク
- 回答者ランク
- 出題者ランク別の1日出題回数制限
- 出題者ランク別の1回配信人数制限
- ワールドの参加枠とシーズン表示

### 安全・通知制御

- 通報
- ブロック
- 通知設定
- quiet hours
- 1日の通知上限
- 深夜通知デフォルトOFF
- 休憩モード

## MVPに含めない機能

| 機能 | MVPで入れない理由 |
| --- | --- |
| タイムライン | クイズ体験よりSNS閲覧が主役になるため。 |
| コメント | モデレーション負荷が増えるため。 |
| DM | 安全設計と通報対応が重くなるため。 |
| フォロー | 初期10人では関係性分岐が不要なため。 |
| ギルド | まず1ワールドの密度を検証するため。 |
| ギルド対抗戦 | ギルド機能の後段にするため。 |
| 課金 | 通知数や配信人数が課金で増える設計を避けるため。 |
| AI自動判定 | MVPでは正誤判定を四択で確実に行うため。 |
| 自由チャット | UGCリスクが高いため。 |
| 公開掲示板 | SNS化と安全運用負荷を避けるため。 |

## 初期登録フロー

1. ユーザーが `/signup` を開く。
2. 18歳以上であることを確認するチェックを入れる。
3. 利用規約に同意する。
4. プライバシーポリシーに同意する。
5. 招待コードまたは管理者承認状態を確認する。
6. world_members の有効メンバー数を確認する。
7. 参加者数が member_limit 未満ならワールド参加を確定する。
8. 参加者数が member_limit 以上なら waitlist へ誘導する。

MVPでは生年月日は保存しない。確認日時として age_confirmed_at を保存する。

## クイズ形式

MVPでは四択クイズを優先する。

| 形式 | MVP対象 | 理由 |
| --- | --- | --- |
| 四択 | 対象 | 正誤判定が明確で、早押し順位と相性がよい。 |
| 短文回答 + 正解別名 | 次段階 | 正規化、別名、表記揺れ対応が必要。 |
| 自由入力のみ | 対象外 | AI判定や曖昧判定に依存しやすい。 |

四択クイズの基本項目は次の通り。

- 問題文
- 選択肢4つ
- 正解選択肢
- 難易度
- カテゴリ
- 出題者
- 公開ステータス

## 出題フロー

1. 出題者が `/create` でクイズを作成する。
2. サーバーが出題者ランクを確認する。
3. 当日の出題回数上限を確認する。
4. 1回で配信できる人数を決定する。
5. 通知対象候補を抽出する。
6. 対象者をランダムまたは重み付きで選ぶ。
7. quiz_launches を作成する。
8. quiz_recipients を作成する。
9. start_at と end_at をサーバーで決定する。
10. 対象者へ通知を送る。
11. start_at から回答受付を開始する。
12. end_at で回答受付を締め切る。
13. 結果と評価を公開する。
14. ランク更新イベントを作成する。

## 回答ルール

- 通知を受け取った人だけが回答対象になる。
- 出題者本人は回答対象外にする。
- 問題表示は通知直後ではなく start_at から開始する。
- start_at はサーバーで決める。
- end_at もサーバーで決める。
- 回答受付は start_at 以降のみ有効にする。
- 回答順位はサーバー受信順で決める。
- 回答者の端末時刻は使わない。
- 1ユーザーが1つの launch に送信できる回答は1件にする。
- 回答後に全回答者の正誤と順位を見られる。

## 結果表示

結果画面では次を表示する。

- 自分の正誤
- 自分の回答順位
- 自分の正解者順位
- 全回答者の回答順位
- 正解者のみの順位
- 正解
- 問題作成者
- 回答数
- 正解率
- クイズ評価ボタン
- 通報ボタン

早い不正解は answer_rank では上位になっても、回答者ランクには大きく加点しない。
回答者ランクに強く効くのは correct_rank と難易度である。

## 通知設定

MVPでは次の通知設定を用意する。

| 設定 | デフォルト | 説明 |
| --- | --- | --- |
| 通知モード | 通常モード | 1日最大5問を目安にする。 |
| quiet hours | 22:00〜8:00 | この時間は通知しない。 |
| 深夜通知 | OFF | 明示的にONにしたユーザーのみ深夜通知を受ける。 |
| 1日の通知上限 | あり | 通知疲れを防ぐ。 |
| 休憩モード | OFF | ONの場合は通知対象から外す。 |

## 管理・安全

MVPで最低限必要な管理機能は次の通り。

- 通報されたクイズの確認
- 通報されたユーザーの確認
- クイズ配信停止
- アカウント停止
- 招待コードの発行と停止
- ウェイトリストの承認
- 参加枠の変更

管理画面をMVP初期から作り込む必要はないが、データモデルと運用手順としては想定しておく。

## 主要な成功指標

MVPでは次を重視する。

- 通知開封率
- 回答参加率
- start_at 後の回答完了率
- クイズ評価平均
- 通報率
- 1人あたりの出題数
- 1人あたりの回答数
- quiet hours や休憩モードの利用率
- ワールド参加者の継続日数
