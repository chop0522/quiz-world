# 通知型早押しクイズワールド 通知設計

## 基本方針

通知はこのゲームの中心体験である。
ただし、通知疲れを起こすとサービス継続率が下がるため、通知量、時間帯、対象者選定を慎重に制御する。

- 通知はゲーム開始の合図である。
- 通知が届いた瞬間に問題開始にはしない。
- start_at をサーバーで決める。
- 通知から15秒後などに開始する。
- カウントダウン画面を用意する。
- 回答受付は start_at 以降にする。
- 回答締切は end_at にする。
- 端末時刻は順位判定に使わない。
- MVP Phase 1は `/home` の15秒ポーリングによる画面内通知で開始する。
- Phase 1.5では `/home` の届いたクイズ一覧、つまり本人宛 quiz_recipients の新着だけRealtime化を検討する。
- Phase 2でWeb Pushを検討する。

## 通知から回答までの体験

1. 通知が届く。
2. ユーザーが通知を開く。
3. `/quiz/[launchId]` に遷移する。
4. start_at までカウントダウンを表示する。
5. start_at になったら問題と選択肢を表示する。
6. ユーザーが回答する。
7. サーバーが回答を受信し、answer_received_at を記録する。
8. answer_rank と correct_rank を決める。
9. end_at 後、または回答後に結果画面へ遷移する。
10. ユーザーがクイズを評価する。

## start_at / end_at

| 項目 | 方針 |
| --- | --- |
| start_at | サーバーで決定する。例: launch作成から15秒後。 |
| end_at | サーバーで決定する。MVP初期値はstart_atから60秒後。 |
| 回答受付 | start_at 以降、end_at 以前のみ有効。 |
| 順位判定 | サーバー受信時刻のみを使う。 |
| 端末時刻 | 表示補助以外には使わない。 |

start_at を置くことで、通知到着タイミングやユーザーの開封タイミングの差をある程度吸収する。

## 通知対象条件

通知対象は、次の条件を満たすユーザーから選ぶ。

- 同じワールドのメンバーである。
- 出題者本人ではない。
- 通知ONである。
- quiet hours 外である。
- 1日の通知上限未満である。
- ブロック関係ではない。
- 直近で活動している。
- 休憩モードではない。
- アカウント停止中ではない。
- 対象クイズや出題者に対する制限がない。

## ブロック関係の扱い

次のいずれかに該当する場合、通知対象から除外する。

- 出題者が対象ユーザーをブロックしている。
- 対象ユーザーが出題者をブロックしている。

ブロックは安全機能であり、ランキング上の不利が出ないように扱う。

## 通知モード

| モード | 1日通知上限 | 説明 |
| --- | ---: | --- |
| 通常モード | 5問 | デフォルト。日常利用向け。 |
| 集中モード | 20問 | 多く参加したいユーザー向け。 |
| 休憩モード | 0問 | 通知なし。 |
| 深夜勢モード | ユーザー設定に従う | 深夜通知を許可するユーザー向け。 |

深夜勢モードでも、本人が明示的に深夜通知を許可していることを必須にする。

## デフォルト設定

| 設定 | デフォルト |
| --- | --- |
| 通知モード | 通常モード |
| quiet hours | 22:00〜8:00 |
| 深夜通知 | OFF |
| 1日の通知上限 | あり |
| 休憩モード | OFF |

深夜通知はデフォルトOFFにする。ユーザーが明示的にONにしない限り、深夜には通知しない。

## 通知送信フロー

1. 出題者がクイズを出す。
2. サーバーが出題者ランクを確認する。
3. 今日の残り出題回数を確認する。
4. 出題者ランクに応じて配信可能人数を決める。
5. 同じワールドのメンバーから対象候補を抽出する。
6. 出題者本人を除外する。
7. 通知OFF、quiet hours、通知上限、休憩モードのユーザーを除外する。
8. ブロック関係を除外する。
9. 直近活動や通知偏りを考慮して、ランダムまたは重み付きで対象者を選ぶ。
10. quiz_launches を作る。
11. quiz_recipients を作る。
12. start_at = now + 15秒 などで設定する。
13. end_at を設定する。
14. 通知を送る。
15. notification_logs に送信結果を記録する。
16. 回答を受け付ける。
17. 結果を公開する。
18. 評価を受け付ける。
19. ランク更新イベントを作成する。

## Phase 1: 15秒ポーリングの画面内通知

MVP Phase 1では、Web Pushではなく画面内通知で体験を検証する。

| 項目 | 方針 |
| --- | --- |
| 更新方式 | `/home` が15秒ごとに `GET /api/quiz-launches` を呼ぶ。 |
| 通知源 | quiz_recipients。本人宛のlaunchのみ表示する。 |
| 表示分類 | 開始前、回答可能、回答済み、終了済み。 |
| 問題表示 | start_at前は問題文と選択肢を表示しない。 |
| Realtime化 | API形を保ったままPhase 1.5でSupabase Realtime化を検討する。 |

`GET /api/quiz-launches` は `/home` 用一覧APIとして定義する。
Phase 1ではこのAPIを15秒間隔で呼び、新着の quiz_recipients と状態変化を取得する。

## Phase 1.5: Realtime化候補

10人テストで15秒ポーリングの反応が遅い、またはクイズ到着体験が弱い場合、Supabase Realtime化を検討する。

Realtime化の対象は `/home` の届いたクイズ一覧に限定する。

- quiz_recipients の本人宛insert/update。
- `/home` 上の新着表示。
- 開始前、回答可能、回答済み、終了済みの状態更新。

Phase 1.5でも、権限と返却フィールドはAPI設計と同じ境界を維持する。

結果画面のRealtime更新、answer一覧のRealtime反映、rankingのRealtime集計はMVP後回しにする。

## Phase 2: Web Push候補

Phase 2でWeb Pushを検討する。

- push_tokens を使う。
- notification_logs に送信結果を残す。
- quiet hours、通知上限、休憩モード、深夜通知設定を送信前に必ず見る。
- iOSやブラウザ依存を10人テスト後に評価する。

## 対象者抽選

MVPでは、最初はシンプルなランダム抽選でよい。
ただし、以下の偏りは避ける。

- 同じユーザーに通知が集中する。
- 特定の出題者からの通知だけが届き続ける。
- 休憩モード後すぐに大量通知が届く。
- 通知上限に近いユーザーを優先し続ける。

将来的には重み付き抽選を検討する。

| 重み付け要素 | 方針 |
| --- | --- |
| 最近通知が少ない | 優先度を上げる。 |
| 最近回答している | 優先度を上げる。 |
| 通知上限に近い | 優先度を下げる。 |
| 同じ出題者から連続 | 優先度を下げる。 |
| 休憩モード解除直後 | 急に増やさない。 |

## 通知失敗時の扱い

| 状態 | 扱い |
| --- | --- |
| sent | 通知成功。 |
| failed | 通知失敗。notification_logs に記録する。 |
| skipped | quiet hours や上限などで送信しなかった。 |
| disabled | トークン無効や通知OFF。 |

通知失敗は回答者ランクのペナルティにしない。

## 結果通知

MVPでは、結果通知は必須ではない。
通知疲れを避けるため、まずは回答後に結果画面へ誘導する。

将来的に結果通知を追加する場合は、次を検討する。

- 自分が正解者順位に入ったときだけ通知する。
- 配信終了時にまとめて通知する。
- 結果通知も1日の通知上限に含めるか分けるかを決める。

## 通知インフラ

| 段階 | 方針 |
| --- | --- |
| Phase 1 | `/home` の15秒ポーリングによる画面内通知。 |
| Phase 1.5 | `/home` の本人宛 quiz_recipients 新着だけSupabase Realtime化を検討。 |
| Phase 2 | Web Pushを検討。 |
| 30〜100人 | 通知キュー、失敗ログ、再送制御を追加。 |
| 本格版 | Expo / React Native + APNs / FCM / Expo Push を検討。 |

PWA通知は、ユーザー許可、ブラウザ、OS、iOSのホーム画面追加などに依存する。
10人で体験を検証してから、ネイティブ化を判断する。

## 通知疲れ対策

- 1日の通知上限を必ず設ける。
- quiet hours をデフォルトで有効にする。
- 深夜通知はデフォルトOFFにする。
- 休憩モードをわかりやすくする。
- 出題者ランクで配信人数を制限する。
- 通知偏りをログで確認する。
- 通知が多すぎるユーザーを抽選で優先しない。
- 通知失敗や未開封をランクで強く罰しない。
